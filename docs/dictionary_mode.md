# Dictionary-Mode Pathway

Mnemosyne's dictionary-mode pathway is the formal channel for languages that
do not yet have reliable full parsing — including dead languages, under-resourced
languages, sign systems, and historical scripts.

The mode is distinct from full morphological analysis and makes no claims about
inflectional structure beyond what a plugin can reliably demonstrate.

---

## Motivation

Full NLP pipelines (spaCy, stanza) exist for a small number of living languages.
For Latin, Ancient Greek, Old English, Sumerian, or Classical Arabic, tools
either do not exist, have significant error rates on real texts, or are not
available as light dependencies.

Dictionary mode lets a plugin emit whatever it *reliably* knows — gloss,
citation form, grammar class, transliteration — without fabricating tense,
mood, case, or dependency structure it cannot support.

> **Rule**: prefer an honest skeleton over a confidently wrong full parse.

---

## What a Dictionary-Mode Plugin Provides

A plugin operating in dictionary mode may emit any subset of the following
fields in `lesson_data`:

| Field | Key | Description |
|-------|-----|-------------|
| Gloss | `gloss` | English meaning(s) |
| Romanization | `romanized` or `pinyin` | Transliteration / phonetic reading |
| Citation form | `citation_form` | Dictionary headword with grammatical summary (e.g. `"amor, amōris m."`) |
| Grammar note | `grammar_note` | Free-text annotation (e.g. `"3rd declension masculine noun"`) |
| Base form | `lemma` | Canonical form when different from the surface token |
| Limitation note | `confidence_note` | Honest description of what the plugin could not do |

The lesson engine's `_build_dictionary` builder gracefully handles any
combination of these fields — missing fields are simply omitted from the
lesson.

### What it does NOT emit

- Conjugation type candidates (`"conjugation"`, `"agreement"`, `"case_agreement"`)
- POS-based multiple-choice drills
- Recognition drills about grammatical correctness
- Any claim that depends on context-sensitive parse output

## Capability Declaration

A dictionary-mode plugin declares its limitations explicitly:

```python
capabilities = LanguageCapabilities(
    code="la",
    display_name="Latin (Classical — dictionary scaffold)",
    direction="ltr",
    script_family="latin",
    tokenization_mode="whitespace",
    morphology_depth="none",            # ← no inflection analysis
    lesson_modes_supported=["dictionary"],  # ← dictionary only
    analysis_depth="dictionary",        # ← citation-form lookup only
    morphology_quality="none",
    syntax_support=False,
    idiom_detection=False,
    transliteration_scheme=None,        # if romanization is not provided
)
```

`lesson_modes_supported=["dictionary"]` is the critical signal: the lesson
route uses `best_lesson_mode()` to select the richest supported mode, so
setting this to `["dictionary"]` ensures every lesson built for this language
uses `_build_dictionary` — never `_build_vocabulary` or `_build_conjugation`.

## Lesson Generation

When `lesson_mode="dictionary"`, `build_lesson()` calls `_build_dictionary`.

### Fields rendered (in order)

1. **Gloss** — English meaning (or provider-supplied fallback).
2. **Romanized** — From `lesson_data["romanized"]` or `["pinyin"]`.
3. **Citation form** — From `lesson_data["citation_form"]`.
4. **Grammar** — From `lesson_data["grammar_note"]`.
5. **Base form** — Shown only when `lemma ≠ display_label`.
6. **Note** — Confidence / provenance note.

### Drills generated

1. **Shadowing** — Always included; the learner reads the token aloud.
2. **Fill-blank** ("What does X mean?") — Only when a gloss is available.

No multiple-choice, no recognition drills. The fill-blank answer is the gloss
string; the learner self-grades (consistent with the rest of the SRS system).

### Explanation prose

Generated by `formatters.dictionary_explanation()`:

| Conditions | Output |
|-----------|--------|
| Gloss known | `"amor" — love, desire.` |
| Language known, no gloss | `"amor" — Latin vocabulary.` |
| Neither | `"amor"` |

---

## Frontend Signal

When `lesson.lesson_mode === "dictionary"`, the lesson modal shows a small
italic label beneath the type badge:

```
[vocabulary]
dictionary lookup — no full parse
```

This is intentionally understated — it informs without alarming. The lesson
content itself is fully usable; the note just prevents the learner from
mistaking dictionary-mode coverage for morphological completeness.

## The Latin Scaffold: Reference Implementation

`backend/plugins/latin.py` is the reference plugin for the dictionary-mode
pathway.

### Design

- **Lexicon**: ~90 common Classical Latin headwords stored as module-level
  tuples: `(citation_form, gloss, grammar_note, pos)`.
- **Lookup key**: macron-normalised, lowercase citation form (so `"amāre"`
  matches the entry keyed on `"amare"`).
- **Token matching**: whitespace-split → strip punctuation → lowercase →
  normalise macrons → dict lookup.
- **Recognised tokens**: emit full `lesson_data` with gloss, citation form,
  grammar note; `confidence=0.85`.
- **Unrecognised tokens** (most inflected forms): emit minimal
  `lesson_data` with `confidence_note` explaining the limitation;
  `confidence=None`.

### What it does NOT do

| Feature | Why absent | Upgrade path |
|---------|------------|--------------|
| Inflected-form lookup | Latin has 100k+ inflected forms; lexicon would be impractical | CLTK morphological analyser |
| POS tagging | Requires morphological context | Stanza Latin model |
| Lemmatisation | Same as above | CLTK / Whitaker's Words |
| Syntax / dependency | Requires full parse tree | CLTK dependency parser |
| Idiom detection | Requires phrase-level lexicon | Phrase dictionary |

### Enabling the plugin

Latin is loaded automatically from `backend/plugins/latin.py` by the plugin
loader.  No extra dependencies are required.

## Adding a New Dead-Language Plugin

Follow the Latin scaffold pattern:

1. Create `backend/plugins/<lang>.py`.
2. Declare capabilities with `morphology_depth="none"`,
   `analysis_depth="dictionary"`, `lesson_modes_supported=["dictionary"]`.
3. Embed a lexicon or point to a lightweight external wordlist.
4. In `analyze_sentence()`, emit `"vocabulary"` candidates with as many of
   the standard dictionary fields as you can reliably supply.
5. Use `confidence_note` to explain what you could not do.
6. Write tests with at least: capability assertions, lexicon coverage,
   known-word extraction, unknown-word graceful handling, and a
   lesson-engine integration round-trip.

### Field conventions

| Language type | `citation_form` example | `grammar_note` example |
|--------------|------------------------|------------------------|
| Latin noun | `"amor, amōris m."` | `"3rd declension masc.; gen. amōris"` |
| Latin verb | `"amō, amāre, amāvī, amātum"` | `"1st conjugation"` |
| Ancient Greek noun | `"λόγος, -ου ὁ"` | `"2nd declension masc."` |
| Old English noun | `"cyning, cyninges m."` | `"strong masculine noun"` |
| Sumerian (transliterated) | `"an"` | `"logogram; sky, heaven; the god An"` |
| Egyptian (transliterated) | `"nṯr"` | `"determinative + consonantal skeleton; god, divine"` |

## Future Languages

The following languages are natural targets for dictionary-mode plugins.
None are committed to any timeline.

### Latin (full upgrade)
- Current scaffold covers ~90 headwords.
- Full upgrade: CLTK morphological analysis, Whitaker's Words integration.
- Upgrade signals: `morphology_depth="shallow"`, `analysis_depth="morphology_light"`.

### Ancient Greek
- Similar to Latin; CLTK supports it.
- Requires polytonic Greek script normalization.
- `transliteration_scheme="beta_code"` or `"ipa"` possible.

### Old English (Anglo-Saxon)
- CLTK supports Old English.
- Strong/weak noun and adjective classes in grammar notes.
- Runic transliteration optional.

### Classical Arabic
- The starter plugin (`backend/plugins/arabic.py`) already provides
  tashkeel-normalised tokenisation.
- Upgrade to dictionary mode with gloss: add `lesson_data["gloss"]` from
  an Arabic–English wordlist.
- Upgrade to morphology: CAMeL-Tools or Stanza-ar.

### Biblical Hebrew
- The starter plugin (`backend/plugins/hebrew.py`) provides nikud-normalised
  tokenisation.
- Full upgrade: Dicta API or a Tanakh lexicon.

### Sumerian
- Cuneiform script requires transliteration (e.g. `"an"`, `"lugal"`).
- No whitespace in cuneiform; tokenisation_mode may need `"segmented"`.
- ePSD (Electronic Penn Sumerian Dictionary) is the reference lexical source.

### Egyptian Hieroglyphic
- Manuel de Codage (MdC) transliteration as the canonical form.
- JSesh font or Unicode hieroglyphs as the display form.
- The lesson engine's `"script"` type is a natural fit for sign entries.
